## Executive Summary

This branch adds Docker socket auto-discovery, clean startup error messages, and an `--open` flag replacing `--no-browser` across all explorers. The core discovery logic is solid and well-tested. ## Important Issues

### [I1] `ensure_sandbox_image()` build errors silently lost
- **File:** `scripts/common.sh:83-97`
- **Bug:** `run_preflight()` calls `report_and_exit` at line 95, then `ensure_sandbox_image` at line 96. If all preflight checks pass, `report_and_exit` returns silently. If the docker build inside `ensure_sandbox_image` then fails, it appends to `ERRORS` — but `report_and_exit` has already run and will never run again. The build failure is never displayed and the launcher continues, hitting a confusing `ImageNotFound` from Python.
- **Impact:** Users see an opaque Python exception instead of the shell-level "Failed to build Docker image" message.
- **Suggested fix:** Move `ensure_sandbox_image` before `report_and_exit`, or call `report_and_exit` a second time after it.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases

### [I2] `DOCKER_HOST` not cleaned up on non-`DockerException`
- **File:** `src/ananta/ananta.py:191-199` and `225-233`
- **Bug:** After setting `os.environ["DOCKER_HOST"]` in Strategy 2 (line 191) and Strategy 3 (line 225), cleanup (`del os.environ["DOCKER_HOST"]`) only runs inside `except DockerException`. If `docker.from_env()` raises anything else — `KeyboardInterrupt`, `MemoryError`, unexpected `ValueError` from the SDK — `DOCKER_HOST` is left permanently set to a non-working socket URL.
- **Impact:** Subsequent code in the same process (or a retry after catching the exception) inherits a stale, broken `DOCKER_HOST`, causing confusing connection errors.
- **Suggested fix:** Use `try/finally` with a success flag, or `os.environ.pop("DOCKER_HOST", None)` in a `finally` block that only fires when the function hasn't returned successfully.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases, Concurrency & State

### [I3] `os.environ` race between concurrent `Ananta` instances
- **File:** `src/ananta/ananta.py:148-246`
- **Bug:** `_check_docker_available()` is a `@classmethod` that mutates `os.environ["DOCKER_HOST"]` (process-global). The `_start_lock` is per-instance (`threading.Lock()` in `__init__`), so two `Ananta` instances calling `start()` on different threads can interleave reads/writes to `DOCKER_HOST`, corrupting each other's discovery.
- **Impact:** Flaky Docker connection failures in multi-instance scenarios (parallel tests, two context managers in threads).
- **Suggested fix:** Either use a class-level lock (`ClassVar[threading.Lock]`) for the env-mutation window, or refactor to use `docker.DockerClient(base_url=socket_url)` directly instead of routing through the environment.
- **Confidence:** High
- **Found by:** Concurrency & State

### [I4] Unvalidated `docker context inspect` stdout written to `DOCKER_HOST`
- **File:** `src/ananta/ananta.py:188-191`
- **Bug:** `result.stdout.strip()` is written directly to `os.environ["DOCKER_HOST"]` with no validation. If `docker context inspect` emits unexpected output (warning prefix, multi-line output, or output from a compromised docker CLI), the value is passed verbatim to the Docker client, potentially redirecting connections.
- **Impact:** Robustness issue in unusual Docker CLI environments; potential security concern if the docker binary on PATH is untrusted.
- **Suggested fix:** Validate the output starts with a known scheme (`unix://`, `tcp://`, `npipe://`) before accepting it.
- **Confidence:** Medium
- **Found by:** Error Handling & Edge Cases, Security

### [I5] Partial `pool.start()` failure orphans already-started containers
- **File:** `src/ananta/ananta.py:461-467`, `src/ananta/sandbox/pool.py`
- **Bug:** In `Ananta.start()`, if `pool.start()` raises partway through (e.g. the third executor fails after two containers started), `self._pool = pool` (line 467) is never reached. `Ananta.stop()` checks `self._pool is not None` — it's `None` — so orphaned containers from the failed pool are never cleaned up.
- **Impact:** Docker container leak on transient startup failure. Repeated retries accumulate orphaned containers.
- **Suggested fix:** Wrap `pool.start()` in try/except that calls `pool.stop()` before re-raising.
- **Confidence:** High
- **Found by:** Error Handling & Edge Cases, Concurrency & State

### [I6] User's original `DOCKER_HOST` silently overwritten by fallback strategies
- **File:** `src/ananta/ananta.py:162-170`
- **Bug:** When `DOCKER_HOST` is set by the user (Strategy 1) and `docker.from_env()` fails, the code falls through to Strategy 2/3 without preserving the original value. If Strategy 2 or 3 succeeds, it overwrites `DOCKER_HOST` with the discovered socket. If Strategy 2 fails, it deletes `DOCKER_HOST` entirely (line 199). Either way, the user's original value is destroyed.
- **Impact:** A user who set `DOCKER_HOST` to a temporarily-unreachable socket (e.g., Docker restarting) will have their environment variable silently overwritten or deleted. On retry, the user's explicit setting is gone.
- **Suggested fix:** Save the original `DOCKER_HOST` at function entry. On success, keep the discovered value. On total failure (Strategy 4), restore the original value.
- **Confidence:** Medium
- **Found by:** Logic & Correctness

### [I7] `web/__main__.py` lacks `parse_args(argv)` extraction — untestable, diverged from siblings
- **File:** `src/ananta/experimental/web/__main__.py:16-24`
- **Bug:** `code_explorer/__main__.py` and `document_explorer/__main__.py` both extract a `parse_args(argv: list[str] | None = None)` function so tests can call `parse_args(["--port", "9000"])` without touching `sys.argv`. `web/__main__.py` inlines `argparse` directly in `main()` with no `argv` parameter, so its argument parsing cannot be unit-tested without subprocess.
- **Impact:** Test coverage gap and code-structure divergence — arguments added to two explorers via the extracted function will be missed in the web explorer's inlined version.
- **Suggested fix:** Extract `parse_args(argv=None)` from `web/__main__.py` to match the pattern in the other two.
- **Confidence:** High
- **Found by:** Contract & Integration

## Suggestions

- **Duplicated `main()` body across all three explorers.** All three `__main__.py` files share an identical launch body (data_dir parsing, state creation, URL printing, Timer-based browser open, uvicorn.run). A `run_explorer(title, create_state_fn, create_api_fn, default_port, argv)` helper in `shared/` would eliminate the duplication and prevent drift. (C5)
- `ContainerPool.start()` `_started` check is not under `self._lock` — safe by current convention (single-threaded caller) but fragile if `with pool:` is ever used from multiple threads. (CS3)
- Several tests (`test_start_retries_after_pool_start_failure`, `test_start_is_idempotent`, etc.) call `start()` without patching `subprocess.run` or setting `DOCKER_HOST`, so they invoke the real docker CLI — may fail in minimal CI environments. (PA4)
- `test_lifespan_prints_clean_error_on_startup_failure` does not assert `SystemExit` code is 1 (wrapped by anyio into `BaseExceptionGroup`). (PA2)
