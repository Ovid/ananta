## Important Issues

### [I1] `try_answer_from_analysis` swallows `PermanentError` (auth failures)
- **File:** `src/shesha/analysis/shortcut.py:154-157`
- **Bug:** `classify_query` (line 105-106) explicitly re-raises `PermanentError` so auth failures surface immediately. But `try_answer_from_analysis` wraps its LLM call in `except Exception: return None` (line 156-157), catching `PermanentError` and silently falling back to the full RLM engine, which will also fail with the same auth error from a different call site.
- **Impact:** Auth failures are swallowed, causing delayed and confusing error presentation. Violates the explicit design intent of commit `d20d641`.
- **Suggested fix:** Add `except PermanentError: raise` before the `except Exception` on line 156, mirroring the pattern in `classify_query`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling

### [I2] `check_updates` endpoint missing `RepoIngestError` catch
- **File:** `src/shesha/experimental/code_explorer/api.py:143`
- **Bug:** `check_updates` calls `state.shesha.create_project_from_repo()` without catching `RepoIngestError`. Both `add_repo` (line 96-99) and the `apply_updates` self-heal path (line 171-174) catch this exception and return HTTP 422.
- **Impact:** Network failures or clone errors during update checks produce a raw 500 error with stack trace instead of a clean 422.
- **Suggested fix:** Wrap line 143 in `try/except RepoIngestError as exc: raise HTTPException(422, detail=str(exc)) from exc`.
- **Confidence:** High
- **Found by:** Error Handling

### [I3] `apply_updates()` call missing `RepoIngestError` catch
- **File:** `src/shesha/experimental/code_explorer/api.py:179`
- **Bug:** `repo_result.apply_updates()` calls `pull()` and `_ingest_repo()`, both of which can raise `RepoIngestError`. This call is not wrapped in a try/except, despite the self-heal path two lines above catching the same exception type.
- **Impact:** Same as I2: unhandled 500 in an API endpoint.
- **Suggested fix:** Wrap line 179 in `try/except RepoIngestError as exc: raise HTTPException(422, detail=str(exc)) from exc`.
- **Confidence:** High
- **Found by:** Error Handling

### [I4] Analysis context injected into RLM question without untrusted boundary wrapping
- **File:** `src/shesha/analysis/shortcut.py:205`
- **Bug:** When the analysis shortcut declines and falls back to the full RLM engine, line 205 concatenates LLM-generated analysis context directly into the question: `rlm_question = f"{analysis_context}\n\n{question}"`. This analysis context (produced by a prior LLM pass over potentially adversarial repo content) is placed in the trusted question field without `wrap_untrusted()` boundary markers. Compare to the shortcut's own path (line 149) which properly wraps the context.
- **Impact:** Indirect prompt injection vector. A malicious repository could craft code comments to manipulate the analysis output, which then flows unwrapped into the RLM system prompt as the "question".
- **Suggested fix:** Wrap analysis context with boundary markers before concatenation, or pass it as a separate parameter to `project.query()` for proper wrapping.
- **Confidence:** High
- **Found by:** Security

### [I5] `Trace.redacted()` drops `metadata` field from steps
- **File:** `src/shesha/rlm/trace.py:88-96`
- **Bug:** The `redacted()` method constructs new `TraceStep` objects with `type`, `content`, `timestamp`, `iteration`, `tokens_used`, and `duration_ms`, but omits the `metadata` field. The `final_source` provenance info (added in commit `78cd760`) and any other step metadata is silently dropped from redacted traces.
- **Impact:** Loss of answer provenance debugging info in redacted traces.
- **Suggested fix:** Add `metadata=step.metadata` to the `TraceStep` constructor in `redacted()`.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I6] `save_sha`/`save_source_url` crash on corrupt `_repo_meta.json`
- **File:** `src/shesha/repo/ingester.py:196,211`
- **Bug:** Both methods call `json.loads(meta_path.read_text())` without catching `json.JSONDecodeError`. If `_repo_meta.json` is corrupted (truncated write, disk error), the unhandled exception prevents SHA/URL from being saved. The same vulnerability exists in `get_saved_sha()` and `get_source_url()`.
- **Impact:** Corrupted metadata file makes the project unusable for update checking.
- **Suggested fix:** Wrap `json.loads()` in `try/except json.JSONDecodeError`, falling back to `{}` for writes and `None` for reads.
- **Confidence:** High
- **Found by:** Error Handling

### [I7] Metadata writes outside try/except after successful ingest
- **File:** `src/shesha/repo/ingester.py:506-514`
- **Bug:** After the main try/except block (which handles cleanup on failure), lines 506-514 call `save_sha()` and `save_source_url()`. If either fails (e.g., corrupt JSON per I6, permissions error), the exception propagates uncaught despite ingestion having already succeeded.
- **Impact:** A successful ingest appears to fail, and metadata loss breaks update checking.
- **Suggested fix:** Move SHA/URL saves inside the try block, or wrap in their own try/except that logs a warning but does not propagate.
- **Confidence:** High
- **Found by:** Error Handling

### [I8] Sandbox `FINAL_VAR` returns empty string for missing variables, bypassing retry
- **File:** `src/shesha/sandbox/runner.py:214`
- **Bug:** `NAMESPACE.get(rv.var_name, "")` returns `""` for missing variables. The engine at line 556 checks `result.final_value` for `None` to detect missing variables, but receives `""` instead. The empty string passes the `is None` check, and the user gets an empty answer instead of a retry prompt.
- **Impact:** Missing sandbox variables silently produce empty answers. Violates CLAUDE.md: "Never let a failed lookup silently produce an empty answer."
- **Suggested fix:** Use `None` as the default: `NAMESPACE.get(rv.var_name)` (returns `None` for missing keys), or `None if rv.var_name not in NAMESPACE else str(NAMESPACE[rv.var_name])`.
- **Confidence:** High
- **Found by:** Logic & Correctness

## Suggestions

- `src/shesha/rlm/trace_writer.py:62,116,157` — `_finalized` flag and file writes are unprotected by a lock; subcall threads calling `write_step()` can race with the main thread's `finalize()`, potentially writing a trace step after the summary line. Add a `threading.Lock`. (Concurrency & State)
- `src/shesha/sandbox/pool.py:60-80` — Overflow container creation (`executor.start()`) runs while holding `self._lock`, blocking all pool operations for seconds during Docker startup. Create the container outside the lock. (Concurrency & State)
- `src/shesha/shesha.py:348` — `check_repo_for_updates` always passes `path=None` to `_handle_existing_project`. The `path` parameter from the original `create_project_from_repo()` is never persisted, so updates ingest the full repo instead of the original scoped subdirectory. (Logic & Correctness)
- `tests/unit/test_shesha.py:1208,1230,1259,1279,1317,1364` — Six occurrences of `shesha_instance._storage` should use the public `shesha_instance.storage` property. (Contract & Integration)
- `tests/unit/test_shesha.py:268` — Test accesses `parser_registry._parsers` private attribute; use `find_parser()` instead. (Contract & Integration)
- `tests/unit/test_shesha.py:316` and `tests/unit/analysis/test_generator.py:18,30` — Tests access `_config` and `_load_prompt` private attributes across module boundaries. (Contract & Integration)
