## Important Issues

### [I1] Code explorer check/apply updates lose subdirectory scope
- **File:** `src/shesha/experimental/code_explorer/api.py:159,195`
- **Bug:** Both `check_updates` and `apply_updates` self-heal call `create_project_from_repo(source_url, name=project_id)` without passing `path`. The fix to load `saved_path` was added to `check_repo_for_updates()` (shesha.py:348), but the code explorer API bypasses that method and calls `create_project_from_repo` directly. Subdirectory-scoped projects expand to the full repo on update.
- **Impact:** Users who ingested a subdirectory of a repo will have their project silently expanded to the full repo on the next update.
- **Suggested fix:** Load the saved path in the code explorer endpoints: `path=state.shesha.repo_ingester.get_saved_path(project_id)` and pass it through. Or have `_handle_existing_project` load the saved path when `path is None`.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [I2] Upload file fully read into memory before size and extension validation
- **File:** `src/shesha/experimental/document_explorer/api.py:176`
- **Bug:** `content = await file.read()` reads the entire upload into memory before the `MAX_UPLOAD_BYTES` check (line 177) and `is_supported_extension` check (line 185). A 500MB upload allocates all that RAM before being rejected. The size limit was added to prevent memory exhaustion, but the check comes too late.
- **Impact:** A malicious or accidental oversized upload can exhaust server memory despite the 50MB limit existing.
- **Suggested fix:** Check extension before reading (`is_supported_extension(file.filename)` only needs the filename). Then use `content = await file.read(MAX_UPLOAD_BYTES + 1)` to cap memory, and check `len(content) > MAX_UPLOAD_BYTES`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling, Contract & Integration, Security

### [I3] Network failure misreported as updates_available
- **File:** `src/shesha/shesha.py:469-473`
- **Bug:** When `get_remote_sha()` returns `None` (network timeout) but `saved_sha` is valid, the code logs a warning but falls through to return `status="updates_available"`. The subsequent `apply_updates()` will also fail with the same network issue, giving the user a confusing experience.
- **Impact:** Transient network failures cause false "updates available" notifications that fail when applied.
- **Suggested fix:** Return a distinct status like `"check_failed"` when `current_sha is None and saved_sha is not None`, or include a `warning` field in the `RepoProjectResult`.
- **Confidence:** High
- **Found by:** Logic & Correctness, Error Handling

### [I4] Overflow container creation holds pool lock during Docker startup
- **File:** `src/shesha/sandbox/pool.py:68-78`
- **Bug:** When the pool is exhausted, `acquire()` creates a `ContainerExecutor` and calls `executor.start()` while holding `self._lock`. Docker container startup takes seconds, during which all `acquire()`, `release()`, and `discard()` calls from other threads are blocked. Under concurrent queries, this creates convoy effects.
- **Impact:** Query latency spikes when multiple queries arrive simultaneously and the pool is exhausted. A finishing query cannot release its executor while a new container is being started.
- **Suggested fix:** Create and start the overflow container outside the lock: (1) check exhaustion under lock, (2) release lock, (3) start container, (4) re-acquire lock, (5) add to `_in_use`.
- **Confidence:** High
- **Found by:** Error Handling, Concurrency & State

### [I5] Shortcut token accumulation uses inconsistent pattern
- **File:** `src/shesha/tui/app.py:515-516`
- **Bug:** `_on_shortcut_answer()` uses `self._cumulative_prompt_tokens += prompt_tokens` while `_on_query_complete()` uses `self._cumulative_prompt_tokens = self._baseline_prompt_tokens + result.token_usage.prompt_tokens`. The `+=` pattern differs from the baseline-reset pattern used everywhere else. Per CLAUDE.md: "Match all user-facing behavior when adding alternate code paths."
- **Impact:** Currently benign (no progress callbacks fire during shortcut), but a maintenance trap if shortcut execution ever gains progress callbacks.
- **Suggested fix:** Use `self._cumulative_prompt_tokens = self._baseline_prompt_tokens + prompt_tokens` for consistency.
- **Confidence:** High
- **Found by:** Logic & Correctness

### [S8] `find_final_answer` Pass 1b may match wrong FINAL occurrence
- **File:** `src/shesha/rlm/engine.py:155-160`
- Pass 1b uses `re.search` which matches the first `FINAL(...)` on any line. If LLM describes its plan ("I'll call FINAL(result)") before the actual call, the wrong content could be captured. Mitigated by Pass 1 (DOTALL) being tried first.
- **Found by:** Logic & Correctness, Error Handling

### [S9] Extension not lowercased in stored format field
- **File:** `src/shesha/experimental/document_explorer/api.py:184`
- `ext = Path(file.filename).suffix` without `.lower()`. Stored `format` could be "TXT" for uppercase extensions. Fix: `ext = Path(file.filename).suffix.lower()`.
- **Found by:** Error Handling

### [S10] `_ingest_repo` always returns `status="created"` even for updates
- **File:** `src/shesha/shesha.py:534`
- The status is hardcoded as `"created"` regardless of `is_update`. API consumers see "created" after applying an update. Cosmetic issue.
- **Found by:** Error Handling

### [S11] CORS `allow_origins=["*"]` with `allow_credentials=True`
- **File:** `src/shesha/experimental/shared/app_factory.py:66-72`

- **File:** `src/shesha/analysis/shortcut.py:115-117`
