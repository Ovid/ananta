## Executive Summary

The test suite has solid behavioral coverage and well-structured test classes, but suffers from two systematic issues: **~140+ in-function imports without explanatory comments** (violating CLAUDE.md's imports-at-top rule) and **~25+ tests accessing private (`_`-prefixed) APIs across module boundaries** (violating CLAUDE.md's API boundaries rule). One mock uses a non-existent field name (`output` instead of `stdout`), and two security-relevant production code paths lack test coverage.

## Important Issues

### [I1] ~140+ in-function imports without explanatory comments across 9+ test files
- **Files:** `tests/unit/rlm/test_engine.py` (~53), `tests/unit/sandbox/test_executor.py` (~32), `tests/unit/rlm/test_trace_writer.py` (~24), `tests/unit/test_shesha.py` (~24), `tests/integration/test_repo_ingestion.py:55-58,75-78`, `tests/unit/analysis/test_shortcut_classifier.py:107,121-123`, `tests/unit/analysis/test_generator.py:62,220`, `tests/unit/tui/test_app.py:585,619,649,690`, `tests/unit/llm/test_client.py:106,125`
- **Bug:** CLAUDE.md states: "All imports must be at the top of the file. If an import cannot be at the top (e.g., circular import, optional dependency), add a comment explaining why." None of these in-function imports have comments. Many are repeated dozens of times (e.g., `find_final_answer` imported inside 25+ test methods in test_engine.py). Some re-import names already at the file top (e.g., `pytest` re-imported at test_shortcut_classifier.py:121).
- **Impact:** Violates a clear project standard. Makes imports harder to audit and creates unnecessary noise.
- **Suggested fix:** Move all in-function imports to the top of each file. Remove duplicate imports.
- **Confidence:** High
- **Found by:** TDD Standards (9 sub-findings)

### [I2] 12 tests call private `_extract_repo_name` directly
- **File:** `tests/unit/test_shesha.py:1220-1311`
- **Bug:** The entire `TestExtractRepoName` class (12 test cases) calls `shesha._extract_repo_name(...)`, a private method. No public equivalent exists. CLAUDE.md: "Never access `_`-prefixed attributes from outside the owning class."
- **Impact:** Tests coupled to internal implementation. If method is renamed/refactored, 12 tests break despite unchanged public behavior.
- **Suggested fix:** Either make `_extract_repo_name` public (it's a pure URL-parsing utility, reasonable to expose), or test through `create_project_from_repo()` by verifying the resulting project_id.
- **Confidence:** High
- **Found by:** Mock & Isolation

### [I3] ~10 tests call/monkey-patch private `_handle_llm_query` and `_resolve_final_var`
- **File:** `tests/unit/rlm/test_engine.py:563,602,629,659,694,1372,1451,1760,1773`
- **Bug:** Multiple tests directly call `engine._handle_llm_query(...)` and `engine._resolve_final_var(...)`. Two tests monkey-patch `engine._handle_llm_query = capturing_handle` (lines 1372, 1451).
- **Impact:** The direct private method calls violate API boundaries. The monkey-patching is particularly brittle -- if the internal method is renamed, tests break silently.
- **Suggested fix:** For subcall tests, test through `engine.query()` with mocked executors that trigger subcalls. For `_resolve_final_var`, test through the full query flow. Replace monkey-patching with `patch.object()` at minimum.
- **Confidence:** High
- **Found by:** Mock & Isolation

### [I4] Private `shesha._storage` access (public property exists)

- **Impact:** CLAUDE.md explicitly calls this out: "The experimental layer uses `Shesha.storage` -- not `_storage`."
- **Suggested fix:** Change `shesha._storage` to `shesha.storage`.
- **Confidence:** High
- **Found by:** Logic & Correctness, TDD Standards

### [I5] Private `shesha._config` access (no public property)
- **File:** `tests/unit/test_shesha.py:346`
- **Bug:** `shesha._config.max_iterations` accesses private `_config`. No public `config` property exists on `Shesha`.
- **Impact:** Violates API boundaries. Unlike I4, there's no existing public property to switch to.
- **Suggested fix:** Either expose a public `config` property on Shesha, or test the config effect indirectly (e.g., verify `shesha.rlm_engine.max_iterations`).
- **Confidence:** High
- **Found by:** Logic & Correctness, Mock & Isolation, TDD Standards

### [I6] Mock uses non-existent `output` field instead of `stdout`
- **File:** `tests/unit/rlm/test_engine_cancellation.py:43`
- **Bug:** `MagicMock(status="ok", output="", final_answer=None)` -- the real `ExecutionResult` dataclass has `stdout`, not `output`. The mock also omits `stderr`, `error`, `final_var`, `final_value`, and `vars`, meaning those will be truthy MagicMock objects if accessed.
- **Impact:** Currently dead setup code (the test's LLM response has no code blocks, so `execute()` is never called). But the wrong field name indicates an interface mismatch that would silently mask bugs if the test were modified to include code blocks.
- **Suggested fix:** Fix to `MagicMock(status="ok", stdout="", stderr="", error=None, final_answer=None, final_var=None, final_value=None, vars=None)`.
- **Confidence:** Medium
- **Found by:** Logic & Correctness

### [I7] Content-Disposition filename sanitization untested (security)
- **File:** `src/shesha/experimental/document_explorer/api.py:304-305`
- **Bug:** `safe_filename = re.sub(r'["\r\n;]', "_", filename)` prevents Content-Disposition header injection. No test verifies this sanitization.
- **Impact:** Security-relevant code path with no regression protection. If removed or broken, header injection becomes possible.
- **Suggested test:** Create a document with filename containing `"` or `\r\n`, call GET download, verify Content-Disposition header has sanitized characters.
- **Confidence:** High
- **Found by:** Coverage Gaps

### [I8] Pool stop-during-overflow race condition untested
- **File:** `src/shesha/sandbox/pool.py:85-89`
- **Bug:** When `pool.stop()` is called while an overflow container is being started (outside the lock), the code checks `if not self._started`, stops the new executor, and raises RuntimeError. This concurrency edge case has no dedicated test.
- **Impact:** A bug in this path could leak containers. The overflow test (`test_overflow_does_not_block_release`) tests the lock-free behavior but not the stop-during-overflow case.
- **Suggested test:** Create pool of size 1, drain it, start overflow acquire in a thread, call `pool.stop()` while overflow `start()` is blocked, assert RuntimeError and container cleanup.
- **Confidence:** Medium
- **Found by:** Coverage Gaps
